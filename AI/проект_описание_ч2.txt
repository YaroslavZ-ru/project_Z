Она продолжает конвейер алгоритма и описывает оставшиеся шаги, обработку ошибок и финальные детали.

```text
### Шаг 4: Агрегация и ранжирование параметров
**Функция:** `aggregate_parameters(candidates, processed_query) -> list[dict]`

**Назначение:**  
Из набора понятий-кандидатов извлечь все возможные параметры, сгруппировать их, вычислить интегральный рейтинг и вернуть ранжированный список (не более 15 параметров).

**Действия:**

1. **Сбор параметров.**  
   Для каждого кандидата из списка `candidates` (полученного на Шаге 3) извлекается массив `parameters`.  
   Каждый параметр в хранилище имеет поля: `name`, `label_ru`, `type`, `description`.  
   На этом этапе мы собираем плоский список всех параметров с указанием `similarity` родительского понятия (кандидата).

2. **Группировка и расчёт частот.**  
   Параметры группируются по полю `name` (уникальное английское имя).  
   Для каждой группы подсчитывается:
   - `freq` — количество кандидатов, в которых встретился этот параметр.
   - `similarities` — список значений `similarity` кандидатов, содержащих данный параметр.
   - `avg_similarity` — среднее арифметическое этих значений.

3. **Вычисление интегрального скора.**  
   Для каждого уникального параметра вычисляется скор по формуле:
   ```
   score = 0.6 * freq_norm + 0.3 * avg_similarity + 0.1 * hint_match
   ```
   где:
   - `freq_norm` — нормализованная частота (частота параметра делится на максимальную частоту среди всех параметров).
   - `avg_similarity` — среднее косинусное сходство понятий, в которых встретился параметр.
   - `hint_match` — степень пересечения токенов из `hints` пользователя с названием параметра (и его русским описанием).  
     Вычисляется как доля токенов подсказок, которые встречаются в `label_ru` или `description` (проверяется лемматизированное вхождение).  
     Если подсказок нет, `hint_match` равен нулю.

4. **Ранжирование.**  
   Параметры сортируются по убыванию `score`.  
   Отбираются первые 15 (или меньше, если всего параметров меньше).  

5. **Добавление уверенности.**  
   В поле `confidence` каждого параметра записывается нормированный скор:  
   `confidence = score / max_score` (если `max_score` > 0, иначе 0).  
   Поле `source` заполняется значением `"knowledge_base"`.

6. **Если параметров меньше 5** – запускается **Шаг 4-бис** (генеративное дополнение), чтобы дополнить список до 5–7 параметров (при условии доступности генеративной модели).

**Результат:**  
Список параметров, готовый для добавления в итоговый ответ.

---

### Шаг 4-бис (опционально): Генеративное достраивание контекста и параметров
**Функция:** `generative_suggest(term, hints, existing_params) -> (list[dict], list[str])`

**Условия включения:**  
Модель запускается только если:
- количество параметров после агрегации < 5, ИЛИ
- в системе явно задан флаг `use_generative = True` (например, через конфигурацию).

Если лёгкая языковая модель не загружена (нет библиотек или памяти), шаг пропускается.

**Действия:**

1. **Загрузка модели (выполняется один раз при старте модуля).**  
   Используется `rugpt3small_based_on_gpt2` (от SberDevices) или аналогичная русскоязычная GPT-подобная модель.  
   Модель должна быть оптимизирована для CPU, занимать не более 500 МБ.  
   Загружается через библиотеку `transformers`.

2. **Формирование промпта.**  
   Создаётся текстовый запрос на русском языке:
   ```
   Термин: {term}. Контекст: {', '.join(hints)}.  
   Перечисли важные параметры, характеристики и ограничения для формального описания этого понятия.
   ```
   Если подсказок нет, строка контекста опускается.

3. **Генерация.**  
   Параметры генерации:
   - `max_new_tokens = 100`
   - `temperature = 0.7`
   - `do_sample = True`
   - `num_return_sequences = 3` (получаем три варианта продолжения).
   Модель возвращает до трёх текстовых строк.

4. **Извлечение параметров из сгенерированного текста.**  
   Используется простой эвристический подход:
   - Каждое предложение разбивается по знакам препинания.
   - Выделяются фразы, похожие на перечисления: содержащие слова «материал», «размер», «тип», «скорость» и т.п.
   - Из фраз извлекается потенциальное имя параметра (ключевое слово) и, если есть, значение или описание.
   - Каждому параметру присваивается:
     - `name` — английская транслитерация или заранее определённый словарь (например, «материал» → `material`).
     - `label_ru` — исходное слово/фраза.
     - `type` — определяется по ключевым словам: если есть указание на единицы измерения (мм, кг, °C) → `float`,  
       если есть слова «наличие», «да/нет» → `boolean`, иначе `string`.  
       Если уверенности нет, ставится `string`.
     - `description` — текст из генерации (очищенный).
     - `confidence` — фиксированное значение 0.5 (так как источник генеративный).
     - `source` — `"generative"`.

5. **Фильтрация.**  
   Исключаются параметры, уже присутствующие в `existing_params` (сравнение по `name`).  
   Отбирается не более 5 дополнительных параметров.

6. **Генерация уточнений.**  
   Дополнительно из того же выхода модели выделяются фразы-рекомендации (например, «желательно указать единицы измерения», «ограничение: только ручной инструмент»).  
   Они помещаются в список `suggested_refinements` (не более 3).

**Возвращаемые значения:**  
- Список новых параметров (может быть пустым).
- Список предложений по уточнению.

---

### Шаг 5: Формирование итогового ответа
**Функция:** `build_response(term, candidates, parameters, suggestions, processed_query, debug_info=None) -> dict`

**Действия:**

1. **Определение доминирующего контекста.**  
   Если есть кандидаты, выбирается домен с максимальной суммой `similarity` (или максимальным средним).  
   Формируется словарь `selected_context`:
   ```
   {
     "domain": str,      # например, "слесарный инструмент"
     "confidence": float # средняя схожесть кандидатов этого домена
   }
   ```
   Если кандидатов нет и использовался резервный режим, домен определяется по ключевым словам запроса, `confidence` выставляется 0.3.

2. **Сборка ответа.**  
   Итоговый словарь:
   ```python
   {
     "status": "ok",
     "term": term,
     "selected_context": ...,
     "parameters": parameters,          # с параметрами из Шага 4 (+4-бис)
     "suggested_refinements": suggestions,
     "warnings": []
   }
   ```
3. **Добавление предупреждений (warnings).**  
   - Если параметров меньше 3: «Слишком мало параметров. Уточните контекст или добавьте подсказки».
   - Если кандидатов не найдено: «Понятие не найдено в базе знаний, параметры предположительные».
   - Если подсказки противоречивы (см. раздел ошибок): «Подсказки имеют низкую семантическую связность, возможен выбор неверного контекста».

4. **Включение отладочной информации.**  
   Если `debug=True` при вызове, в ответ добавляется ключ `debug_info`:
   ```python
   {
     "query_vector": list,          # итоговый вектор запроса
     "candidates_raw": list,        # полный список кандидатов до фильтрации
     "scores_distribution": list    # значения скоров параметров до нормировки
   }
   ```
   Эта информация помогает при тестировании и доработке алгоритма.

5. **Возврат итогового словаря.**

## ПОЛНАЯ ОБРАБОТКА ОШИБОК И ГРАНИЧНЫЕ СЛУЧАИ

| Ситуация | Поведение |
|----------|-----------|
| `term` после очистки — пустая строка | Возвращается `{"status": "error", "message": "Пустой термин. Введите значимое слово."}` |
| fastText модель не загрузилась (файл `cc.ru.300.bin` отсутствует) | Аварийный режим: попытаться загрузить статические эмбеддинги из резервного numpy-файла. Если и его нет — вернуть ошибку: `{"status": "error", "message": "Модель эмбеддингов недоступна."}`. |
| База знаний (JSON/SQLite) пуста или повреждена | Работа в режиме «только генерация» (если доступна модель) или выдача шаблонных параметров. В `warnings` добавляется "База знаний недоступна". |
| Таймаут операции поиска (установлен лимит 2 секунды) | Прерывание поиска. Возврат промежуточных результатов с уведомлением: `{"status": "partial", "message": "Поиск прерван по таймауту..."}`. |
| Противоречивые подсказки: косинусное сходство между векторами подсказок < 0.1 | В `warnings` добавляется "Подсказки слабо связаны. Возможна неточность контекста." Алгоритм продолжает работу. |
| Кандидатов нет после понижения порога до 0.2 | Активируется резервный режим (шаблоны или генерация). |
| Генеративная модель выбросила исключение или выдала нечитаемый текст | Параметры модели игнорируются. В `warnings` сообщение: "Генеративное дополнение не удалось." |

## ПРИМЕР РАБОЧЕГО ЦИКЛА

**Вход:**
```json
{
  "term": "ключ",
  "hints": ["техника", "вращение"],
  "debug": false
}
```

**Пошаговое выполнение:**

1. **Предобработка:**  
   term = "ключ", hints = ["техника", "вращение"]  
   Лемматизация: ["ключ", "техника", "вращение"]  
   Синонимы: ключ → "инструмент" (вес 0.4), техника → "механизм" (вес 0.4), вращение → "поворот" (вес 0.4).  
   Расширенные токены с весами готовы.

2. **Векторизация:**  
   V_term = fasttext["ключ"]  
   V_hints = (fasttext["техника"] + fasttext["вращение"]) / 2  
   V_syn = (fasttext["инструмент"] + fasttext["механизм"] + fasttext["поворот"]) / 3  
   V = 0.7*V_term + 0.3*V_hints + 0.1*V_syn  
   L2-нормализация.

3. **Поиск в базе знаний:**  
   Найдено 12 кандидатов с косинусным сходством от 0.88 до 0.32.  
   Самые близкие: "ключ гаечный" (0.88), "ключ разводной" (0.82), "ключ динамометрический" (0.79)...  
   Доминирующий домен: "слесарный инструмент".

4. **Агрегация параметров:**  
   Из кандидатов извлечены параметры: "материал", "размер", "форма головки", "момент вращения", "тип привода" и т.д.  
   После группировки и расчёта скоров топ-5:  
   - размер (score 0.95)  
   - материал (0.92)  
   - форма головки (0.88)  
   - момент вращения (0.76)  
   - тип привода (0.71)  

   Параметров достаточно (>5), генеративная достройка не запускается.

5. **Формирование ответа:**  
   ```python
   {
     "status": "ok",
     "term": "ключ",
     "selected_context": {"domain": "слесарный инструмент", "confidence": 0.88},
     "parameters": [ ... ],
     "suggested_refinements": [
       "Исключите электрические и пневматические ключи, если рассматривается ручной инструмент",
       "Укажите диапазон размеров (мм) и момент вращения (Н·м)",
       "Можно добавить параметр 'покрытие' (хромирование, фосфатирование)"
     ],
     "warnings": []
   }
   ```

## ТРЕБОВАНИЯ К СРЕДЕ И ЗАВИСИМОСТИ

**Python:** версия 3.10 или выше.

**Обязательные библиотеки:**
- `numpy` — работа с векторами.
- `fasttext` — загрузка модели `cc.ru.300.bin`.
- `pymorphy2` — лемматизация и морфологический анализ.
- стандартные модули: `json`, `re`, `sqlite3`.

**Опциональные библиотеки (для расширенных возможностей):**
- `faiss-cpu` — для быстрого поиска по большой базе (>10 000 понятий). Устанавливается отдельно.
- `transformers`, `torch` — для генеративного компонента (модель `rugpt3small_based_on_gpt2` или `DistilRuBERT`). Не обязательны, если генерация не используется.

**Файлы данных:**
- Предобученная fastText модель: скачать `cc.ru.300.bin.gz` с https://fasttext.cc/docs/en/crawl-vectors.html, распаковать, разместить в директории `models/`.
- Статический словарь синонимов (JSON) — должен быть подготовлен или загружен.

**Модели для генерации:**  
- `rugpt3small_based_on_gpt2` (или аналогичная) скачивается автоматически при первом вызове через `transformers`.

## ДОПОЛНИТЕЛЬНЫЕ ТЕХНИЧЕСКИЕ ТРЕБОВАНИЯ И УТОЧНЕНИЯ

### 1. Альтернативные векторные модели (fallback-стратегия)
Основная модель: **fastText `cc.ru.300.bin`** (CBOW, 300 dim, character n-grams).
Резервные варианты (если основная недоступна):
- **word2vec RusVectores** (`rusvectores.org/ru/models/`) — формат `*.bin` или `*.vec`, размерность 300.
- **RuBERT embeddings** (через `transformers`) — только для генеративного режима, требует больше ресурсов.
- Статические эмбеддинги в формате `numpy.memmap` для экономии памяти.

**Приоритет загрузки:**
```
fastText → word2vec → RuBERT (только генерация) → аварийный режим (шаблоны)
```

### 2. Детализация словаря синонимов
- Формат хранения: JSON с ключом-леммой и списком синонимов с весами.
```json
{
  "ключ": [
    {"syn": "инструмент", "weight": 0.4},
    {"syn": "отмычка", "weight": 0.3}
  ]
}
```
- Источник: **RuWordNet** (http://ruwordnet.ru/) или агрегированный словарь из OpenCorpora + ABBYY Finetune.
- Предварительная лемматизация ключей словаря через `pymorphy2` перед загрузкой.
- Кэш словаря в памяти: `Dict[str, List[Tuple[str, float]]]`.

### 3. Схема базы знаний (SQLite пример)
```sql
CREATE TABLE concepts (
    id TEXT PRIMARY KEY,
    term TEXT NOT NULL,
    domain TEXT,
    embedding BLOB,  -- 300 float32, packed
    created_at DATETIME
);

CREATE TABLE parameters (
    id INTEGER PRIMARY KEY,
    concept_id TEXT REFERENCES concepts(id),
    name TEXT,        -- snake_case, en
    label_ru TEXT,    -- человекочитаемое, ru
    type TEXT CHECK(type IN ('string','integer','float','boolean','enum')),
    description TEXT,
    unit TEXT,        -- опционально: мм, Н·м, °C
    enum_values TEXT  -- JSON-массив для type='enum'
);

CREATE INDEX idx_concepts_embedding ON concepts(domain);
```

### 4. Управление конфигурацией
Вынести настраиваемые параметры в отдельный файл `config.json` или dataclass:
```python
@dataclass
class Config:
    min_confidence: float = 0.3
    max_candidates: int = 20
    max_parameters: int = 15
    use_generative: bool = False
    generative_model: str = "rugpt3small_based_on_gpt2"
    timeout_seconds: float = 2.0
    cache_embeddings: bool = True
    log_level: str = "INFO"
```

### 5. Логирование и отладка
- Использовать стандартный `logging` с уровнями: `DEBUG` (векторы, скоры), `INFO` (ход выполнения), `WARNING` (пороги, резервные режимы), `ERROR` (критические сбои).
- При `debug=True` в ответ добавлять:
  - Нормализованный вектор запроса (первые 10 значений для краткости).
  - Топ-5 кандидатов с их `similarity`.
  - Распределение скоров параметров до нормировки.

### 6. Требования к памяти и производительности
| Компонент | Примерный объём | Примечание |
|-----------|----------------|------------|
| fastText `cc.ru.300.bin` | ~3.5 ГБ (в памяти) | Использовать `fasttext.load_model()` с memory-mapping |
| FAISS индекс (10k понятий) | ~20–50 МБ | Зависит от квантования |
| Словарь синонимов | <10 МБ | Загружается целиком в dict |
| Генеративная модель (опционально) | ~500 МБ | Только `rugpt3small`, CPU-оптимизированная |

**Рекомендации:**
- Загружать тяжелые модели лениво (при первом вызове соответствующей функции).
- Кэшировать векторы запросов при повторных обращениях (LRU-кэш на 100–500 записей).
- Для production: вынести модели в отдельный микросервис с gRPC/REST.

### 7. Обработка пользовательского ввода (безопасность)
- Очистка `term` и `hints` от управляющих символов, инъекций через `re.sub(r'[^\w\s\-]', '', text)`.
- Ограничение длины: `term` ≤ 100 символов, каждый `hint` ≤ 50 символов.
- Для генеративного режима: экранирование входных данных перед передачей в модель (защита от prompt-injection).

### 8. Тестирование и метрики качества
**Юнит-тесты (pytest):**
- Валидация входных данных (пустой термин, недопустимые символы).
- Корректность лемматизации и расширения синонимами.
- Воспроизводимость векторизации (детерминированность при фиксированном seed).

**Интеграционные тесты:**
- Полный цикл: вход → ответ (проверка структуры, типов, диапазонов `confidence`).
- Резервные режимы: эмуляция отсутствия модели/БД.

**Метрики качества (для оценки разработчиком):**
- `Precision@5`: доля релевантных параметров в топ-5.
- `Context Accuracy`: корректность определения домена (`selected_context`).
- `User Satisfaction`: опционально, через A/B-тестирование интерфейса.

### 9. Механизм обновления базы знаний
- Добавление нового понятия:
  1. Валидация термина и домена.
  2. Векторизация через ту же `fastText`-модель (консистентность!).
  3. Сохранение в БД + перестройка FAISS-индекса (если используется).

### 13. Уточнение: Векторизация понятия при добавлении в базу знаний

Для обеспечения консистентности поиска эмбеддинг понятия должен вычисляться по той же логике, что и вектор запроса, но без пользовательских подсказок.

**Алгоритм:**
1. Взять термин понятия, лемматизировать (`pymorphy2`), разбить на токены.
2. Расширить каждый токен синонимами из того же статического словаря, который используется в Шаге 1, с весом 0.4 (только для токенов самого термина).
3. Для каждого токена (исходные + синонимы) получить 300-мерный вектор через `fastText`.
4. Вычислить итоговый вектор как взвешенное среднее:
   - Исходные токены термина — вес 0.7 (распределяется равномерно, если их несколько).
   - Все синонимы — суммарный вес 0.3 (равномерно по всем синонимам).
   Если термин состоит из одного слова и синонимов нет, вектор просто равен `fastText[лемма]`.
5. L2-нормализовать полученный вектор.
6. Сохранить нормализованный вектор в поле `embedding` записи понятия.

Это гарантирует, что при поиске запрос «ключ» + «техника» будет корректно сравниваться с понятием «ключ гаечный», потому что их векторы построены в одном семантическом пространстве с аналогичным учётом синонимов.

### 14. Механизм сессии для интерактивного уточнения
**Функция:** `update_session(session_id: str, term: str, new_hints: list[str], previous_response: dict) -> dict`
**Назначение:** Обеспечить накопление контекста и пошаговое уточнение понятия без потери состояния.
**Логика:**
- Сессия хранится в памяти (LRU-кэш, TTL=30 мин) или в лёгкой SQLite-таблице `sessions`.
- Поля сессии: `term` (фиксируется при первом вызове), `accumulated_hints` (список всех уточнений), `selected_domain` (выбранный пользователем домен), `history` (стек ответов для отката).
- При новом запросе: `final_hints = session.accumulated_hints + new_hints`. Запускается основной конвейер с `final_hints`.
- Если пользователь явно указал домен, поиск на Шаге 3 фильтруется строго по `domain`.
- Возвращается обновлённый ответ + текущее состояние сессии для передачи на клиент.

### 15. Структурированные ограничения (constraints)
В выходную схему ответа добавляется поле `constraints`:
```python
"constraints": [
  {
    "type": "exclude" | "require" | "range",
    "parameter": str,      # имя параметра из parameters
    "value": Any,          # строка, число или [min, max]
    "description": str     # человекочитаемое пояснение
  }
]
```
**Правила генерации (эвристика + NLP):**
- Фразы «только ручной», «без электричества» → `{"type": "require", "parameter": "power_source", "value": "manual", ...}`
- Фразы «размер от 10 до 20» → `{"type": "range", "parameter": "size_mm", "value": [10.0, 20.0], ...}`
- Ограничения накладываются на финальный список параметров перед отдачей клиенту и учитываются при сохранении понятия.

### 16. Функция сохранения понятия в БЗ
**Функция:** `save_concept(term: str, domain: str, parameters: list[dict], constraints: list[dict], relations: list[dict] = None) -> dict`
**Действия:**
1. Валидация: проверка уникальности `(term, domain)`, наличие обязательных полей у параметров.
2. Векторизация термина по той же логике, что в Шаге 2 (без user hints, только термин + синонимы).
3. Запись в таблицу `concepts` → вставка параметров в `parameters`.
4. Если указаны `constraints` → сохранение в новую таблицу `concept_constraints`.
5. При использовании FAISS → инкрементальное добавление вектора в индекс.
6. Возврат: `{"status": "ok", "concept_id": str, "warnings": [...]}`.

### 17. Таблица связей между понятиями (онтология)
```sql
CREATE TABLE relations (
  id INTEGER PRIMARY KEY,
  source_concept_id TEXT REFERENCES concepts(id) ON DELETE CASCADE,
  target_concept_id TEXT REFERENCES concepts(id) ON DELETE CASCADE,
  relation_type TEXT CHECK(relation_type IN ('is_a', 'part_of', 'related_to', 'synonym')),
  confidence REAL,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
CREATE INDEX idx_relations_source ON relations(source_concept_id);
```
**Использование в алгоритме:**
- На Шаге 3, после первичного поиска, можно расширить пул кандидатов через `related_to` или `synonym` связи с понижением веса на `0.5` за каждый шаг по графу.
- На Шаге 5 в ответ добавляется поле `related_terms` (топ-3 связанных понятия).

### 18. Механизм выбора домена при омонимии
Если на Шаге 3 найдено ≥2 домена с `similarity > 0.7` и разница между лучшими кандидатами < 0.1:
- Вместо `selected_context` возвращается `context_candidates`:
```python
"context_candidates": [
  {"domain": "слесарный инструмент", "confidence": 0.88},
  {"domain": "музыкальная теория", "confidence": 0.85}
]
```
- В `suggested_refinements` добавляется: «Уточните контекст: выберите домен или добавьте тематическую подсказку».
- При повторном вызове с указанием `target_domain` алгоритм сужает поиск до выбранной области.

### 19. Безопасность генеративного режима
Перед передачей данных в языковую модель (Шаг 4-бис) применяются жёсткие фильтры:
1. Регулярная очистка: `re.sub(r'[<>"{}\\|;`]', '', text)` для `term` и `hints`.
2. Запрет на системные инструкции: если вход содержит `</>`, `###`, `system:`, `ignore previous`, `выполни код` → возврат `{"status": "error", "message": "Запрос отклонён: недопустимые инструкции."}`.
3. Ограничение длины контекста для модели: ≤ 500 символов.
4. Пост-фильтрация выхода модели: исключение строк, содержащих исполняемый код, URL или markdown-разметку команд.
5. Логирование всех отклонённых запросов в таблицу `security_log` для анализа.
